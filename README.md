# Payment Reconciliation — Тестовое задание

Система для сверки банковских платежей со счетами. 
Бэк на FastAPI, фронт на React, данные в памяти.

## Как запустить

### Backend
```bash
cd backend
python -m venv venv
venv\Scripts\activate  # Windows
pip install -r requirements.txt
uvicorn main:app --reload
```
Swagger UI: http://localhost:8000/docs

### Frontend
```bash
cd frontend
npm install
npm run dev
```
Открыть: http://localhost:5173

## Тестовые выписки

Специально подготовил три файла чтобы показать разные сценарии:

- `sample_statement.csv` — основная выписка, 3 транзакции
- `statement_duplicate.csv` — та же выписка повторно, показывает идемпотентность (план будет пустым)
- `statement_partial.csv` — частично новая выписка, TX001 пропускается, TX004 и TX005 обрабатываются

**Сценарий для проверки идемпотентности:**
1. Загрузи `sample_statement.csv` → подтверди план
2. Загрузи `statement_duplicate.csv` → увидишь "Все транзакции уже обработаны"

**Сценарий для проверки частичной догрузки:**
1. Загрузи `sample_statement.csv` → подтверди план
2. Загрузи `statement_partial.csv` → в план попадут только новые TX004 и TX005


## Допущения

- Данные живут пока запущен сервер, при перезапуске сбрасываются
- Номера счетов в тексте — четырёхзначные числа начиная с 4 (4521, 4522...)
- Валюту не конвертирую, сравниваю только одинаковые
- Оператор — просто строка с именем, авторизации нет
- "Разделить платёж" трактую как привязку одного платежа к нескольким 
  счетам при условии что суммы совпадают. Ручное дробление суммы 
  оператором не реализовано

## Что сделано

- Матчинг по IIN/BIN клиента + номер счёта из purpose_text + сумма
- Три статуса: AUTO / UNCERTAIN / UNMATCHED с объяснением почему
- Confidence score от 0 до 1 — показывает насколько система уверена в матче
- Один платёж → несколько счетов (TX003 автоматически покрыл inv-4522 и inv-4523)
- Идемпотентность — повторная загрузка той же выписки не создаёт дубли
- Частичная догрузка — обрабатываются только новые tx_id
- Оператор может вручную поменять привязку счёта через кнопку "Изменить"
- Оператор может отметить платёж как аванс
- После подтверждения статусы счетов обновляются на PAID
- Audit log — кто, когда, какие транзакции к каким счетам привязал  

## Что не реализовано

- Drag & drop между колонками
- Ручное дробление суммы платежа между счетами — 
  требует уточнения бизнес-требования
- Автоматическое выставление статуса OVERDUE по due_date

## Trade-offs

**In-memory вместо БД** — проще запустить и проверить, для демо достаточно.
В проде однозначно нужна БД с транзакциями чтобы не было рассинхрона.

**Regex для поиска номеров счетов** — быстро и предсказуемо. 
Fuzzy search был бы мощнее но избыточен если номера всегда одного формата.

**plan_id при подтверждении** — оператор работает со снимком данных на момент загрузки выписки.
Это защищает от ситуации когда кто-то меняет счета пока оператор смотрит на план.
